# AI 코딩 협업 사용 패턴 분석

## 1. 문서 목적
이 문서는 한 사용자가 처음으로 AI 코딩 협업을 사용할 때, 어떤 사고과정으로 요청을 만들고, 어떤 순서로 문제를 좁혀가며, 어떤 방식으로 AI를 "설계자"가 아니라 "실행 파트너"처럼 활용했는지를 분석한 문서다.

핵심은 코드 수정 내역이 아니다.  
핵심은 아래 3가지다.

1. 어떤 문제를 먼저 꺼냈는가
2. 어떤 근거를 가지고 요구사항을 계속 정제했는가
3. AI에게 어떤 단위로 일을 맡기면 실제 운영 도구가 만들어지는가

이 문서는 "나처럼 AI에게 일 시키는 법"을 다른 사람에게 공유하기 위한 관찰 기록으로 읽으면 된다.

## 2. 한 줄 요약
이 사용 방식의 본질은, AI에게 처음부터 완성된 상세 명세를 주는 것이 아니라, 실제 운영 중 부딪히는 문제를 하나씩 던지고, 그때그때 로그와 증거를 붙여가며, 도구를 운영형 제품으로 성장시키는 방식이다.

조금 더 풀어서 말하면 다음과 같다.

- 처음에는 "무엇을 만들지"보다 "지금 막히는 운영 문제"를 먼저 던진다.
- AI가 뭔가 만들면 바로 실제 환경에서 돌려본다.
- 돌려본 결과를 로그, 스크린샷, 수동 명령 결과로 되돌려준다.
- 그러면 요구사항이 더 선명해지고, 그 요구사항이 다시 도구 구조로 흡수된다.
- 이 과정을 반복하면서 스크립트가 "한 번 쓰는 자동화"가 아니라 "다음 사람도 돌릴 수 있는 운영 엔진"으로 바뀐다.

## 3. 전체 흐름 개요
전체 대화는 대략 아래 순서로 진행되었다.

1. 실행 환경 연결
2. 바로 필요한 기능 수정
3. 실제 장애를 만나며 디버깅
4. 옵션, 경로, 설정 구조 정리
5. 중복 실행 방지, nohup, lock, stop, 로그 등 운영 안정화
6. status board, fleet status, result log 같은 관측 체계 추가
7. import 원격 실행, CSV 매핑, 타깃 노드 표준화
8. README/SOP/CONFIG 문서화
9. 프로젝트 이름/스크립트 이름/브랜딩 정리
10. 마지막에는 "이 도구를 만든 방식 자체"를 다시 분석하려는 메타 단계로 이동

중요한 점은, 이 흐름이 처음부터 설계된 것이 아니라는 점이다.  
실행하다 보니 다음 요구가 생겼고, 그 요구가 쌓이면서 구조가 만들어졌다.

## 4. 가장 중요한 특징: 문제 중심으로 시작했다
이 사용자는 AI에게 "새로운 시스템 하나 설계해줘"라고 시작하지 않았다.  
대신 아주 구체적인 운영 문제부터 던졌다.

예를 들면 이런 식이다.

- git 연결
- xml 저장 경로 변경
- conf로 빼기
- run 시 virsh에 VM 있으면 막기
- lv down 실패 원인 보기
- import는 타깃에서 실행하게 바꾸기
- 로그가 왜 안 남는지 보기
- status 보드가 왜 이상한지 보기

이 방식의 장점은 명확하다.

- 처음부터 거대한 설계를 하지 않아도 된다.
- 당장 필요한 것부터 해결되므로 체감 가치가 빠르다.
- 잘못된 방향으로 크게 가기 전에 바로 수정할 수 있다.

즉, 이 사용자는 AI를 "요구사항 분석가"보다 "즉시 실행 가능한 운영 엔지니어"처럼 썼다.

## 5. 사고과정의 핵심: 완성형 요구사항이 아니라 진화형 요구사항
이 사용자는 처음부터 완벽한 명세를 주지 않았다.  
대신 한 번 실행해보고, 거기서 보이는 불편과 실패를 다음 요구사항으로 바꿨다.

이 패턴은 계속 반복된다.

1. 기능을 하나 요청한다.
2. 바로 실환경에서 돌린다.
3. 예상과 다르게 동작하면 "증상"을 다시 던진다.
4. 그 증상 때문에 필요해진 제약조건을 추가한다.
5. 이전보다 더 운영 친화적인 구조로 수정하게 만든다.

예를 들어 사고 흐름은 이런 식으로 읽힌다.

- "xml은 여기 저장되면 안 된다"
- "그럼 conf로 빼자"
- "conf도 지금 위치는 아니다"
- "옵션 이름도 직관적이어야 한다"
- "nohup으로 돌아야 한다"
- "중복 실행되면 안 된다"
- "죽으면 lock은 어떻게 하냐"
- "stop도 편해야 한다"
- "로그는 트러블슈팅 가능해야 한다"
- "상태판은 여러 대를 한 번에 봐야 한다"

즉, 요구사항은 처음부터 기능 중심이 아니라, 사용하면서 불편했던 지점이 계속 구조화된 것이다.

## 6. 요청 방식의 큰 특징: 구현 방법보다 원하는 동작을 먼저 말했다
이 사용자는 자주 "어떻게 구현해라"보다 "어떻게 동작해야 하느냐"를 먼저 말했다.

대표적인 표현은 이런 류다.

- "그러지 말고 /data 밑에 넣자"
- "conf로 만들어"
- "중복 실행되지 않게"
- "타깃에서 실행해야 한다"
- "로그는 자세히 남아야 한다"
- "한 번에 보는 현황판"
- "기본은 이거고, 옵션 주면 저렇게"

이 방식은 AI 활용에서 매우 중요하다.  
왜냐하면 구현 세부사항에 먼저 묶지 않으면서도, 원하는 결과는 명확히 고정하기 때문이다.

즉, 사용자는 AI에게 자유를 주되, 결과는 강하게 통제했다.

## 7. 또 다른 특징: 말은 짧았지만 기준은 매우 선명했다
표면적으로는 짧고 툭툭 던지는 말이 많았다.  
하지만 실제로는 기준이 계속 분명하게 드러났다.

예를 들면 다음과 같은 기준이다.

- 경로는 운영 표준(`/data`)을 따라야 한다.
- import는 소스가 아니라 타깃에서 실행되어야 한다.
- conf는 장황하면 안 되고 단순해야 한다.
- 옵션 이름은 기술적이기보다 직관적이어야 한다.
- 로그는 "실패했다"가 아니라 "왜 실패했는지"가 보여야 한다.
- status는 개별 로그가 아니라 한 화면에서 판단 가능해야 한다.
- 문서는 실제 따라 할 수 있게 쓰여야 한다.

즉, 문장이 짧다고 해서 기준이 없는 것이 아니었다.  
오히려 긴 설명 대신 운영 원칙이 명확히 머릿속에 있었고, 그것을 계속 보정하는 방식이었다.

## 8. 운영 중 증거를 계속 붙인 점이 매우 중요했다
이 사용 방식에서 가장 강력한 부분 중 하나는 "말로만 설명하지 않고 증거를 계속 붙였다"는 점이다.

증거 유형은 다양했다.

- 터미널 스크린샷
- 수동 명령 결과
- 에러 메시지 원문
- 로그 tail
- 특정 XML 조각
- qemu-img info 결과
- 현재 브랜치와 메인 브랜치 차이에 대한 관찰

이 증거들은 단순 참고가 아니었다.  
요구사항을 더 정확하게 만드는 재료였다.

예를 들면:

- "lv down 실패"는 단순 기능 추가 요청이 아니라 실제 운영 장애였다.
- "맥 중복"은 이론이 아니라 import 실패 로그로 확인됐다.
- "virtio-scsi 문제"는 XML과 import 결과 비교를 통해 좁혀졌다.
- "MAX_JOBS가 4로 고정된 것 같다"는 status/진행률 관측에서 나온 문제였다.

즉, 사용자는 AI를 상대로 추상 토론을 하지 않았고, 거의 항상 실제 증거를 기반으로 대화를 전진시켰다.

## 9. 요청의 기본 단위는 "작은 변경 + 즉시 검증"이었다
전체 대화를 보면, 기능을 한 번에 크게 몰아넣는 경우보다 작은 변경을 자주 던졌다.

예를 들면:

- 경로 하나 바꾸기
- 옵션 하나 추가하기
- 로그 형식 하나 고치기
- 컬럼 하나 바꾸기
- 스크립트 이름 바꾸기
- status 보드에 host 추가하기

이 방식은 AI 협업에서 매우 유리하다.

- 무엇 때문에 좋아졌는지/나빠졌는지 알 수 있다.
- 실패했을 때 원인 범위가 작다.
- 수정 후 바로 검증하기 쉽다.

즉, 사용자는 자연스럽게 "작게 자르고, 바로 돌려보고, 다시 요구한다"는 개발 리듬을 만들었다.

## 10. 기능 추가보다 운영 안정화를 더 중요하게 봤다
초반에는 기능 수정이 많아 보이지만, 시간이 갈수록 진짜 중심은 기능보다 운영 안정화로 이동한다.

중반 이후에 반복적으로 등장한 주제는 아래와 같다.

- nohup
- stop
- lock
- stale lock
- 중복 실행 차단
- 실패 로그 tail
- result.log
- status_board
- 여러 서버 한 번에 보기
- 원격 실행 실패 시 로그 남기기

이건 매우 중요한 포인트다.  
처음 AI를 쓰는 사람들은 종종 "새 기능 추가"에만 집중하는데, 이 사용자는 실제 운영 도구가 되려면 반드시 필요한 요소들을 빠르게 붙였다.

즉, 스크립트를 자동화 도구로 끝내지 않고 운영 제품으로 밀어 올린 것이다.

## 11. 구조를 복잡하게 만들기보다 다시 단순화하려는 성향이 강했다
대화 중간중간 사용자는 계속 복잡도를 줄이려고 했다.

대표적인 표현은 이런 흐름이다.

- ova, v2v 폴더를 굳이 나눌 필요 있나
- conf는 더 단순해야 하지 않나
- import.conf는 굳이 따로 필요 없다
- 스크립트 둘 중 하나만 남기자
- 로그도 너무 많으면 보기 힘들다
- 결과는 하나로 모아라
- status는 VM만 보여라

이것은 매우 중요한 사용 패턴이다.  
AI는 요청을 받으면 종종 구조를 늘려서 문제를 풀려고 한다.  
이 사용자는 그때마다 "운영자가 실제 쓸 때 복잡한가?"를 기준으로 다시 쳐냈다.

즉, 추가만 잘한 것이 아니라 삭제와 단순화도 잘했다.

## 12. 기술적 정확성보다 운영 맥락의 정확성을 우선했다
대화 전체를 보면, 질문은 종종 기술적으로 매우 세부적인 방향으로 갔다.

예를 들면:

- raw/qcow2 의미 차이
- input format 해석
- virtio vs virtio-scsi
- controller/serial
- MAC pool
- libguestfs / virt-v2v precheck

하지만 이 질문들의 목적은 학술적 이해가 아니었다.  
항상 "그래서 지금 운영에서 뭘 바꿔야 하느냐"로 되돌아왔다.

즉, 이 사용자는 기술 질문을 해도 결국은 아래 방향으로 귀결시켰다.

- 이 옵션을 기본값으로 둘까?
- 이 로직은 제거할까?
- 수동으로 해야 하나 자동으로 해야 하나?
- 문서에는 뭐라고 적어야 하나?

이 점이 중요하다.  
AI와 협업할 때 기술 논의가 길어져도, 결국 행동 가능한 결정으로 돌아와야 진전이 난다.

## 13. 언제 문서화를 시작했는지도 의미가 있다
문서화 요청은 처음부터 나오지 않았다.  
어느 정도 실제 도구가 돌아가기 시작한 뒤에 문서화 요구가 강해졌다.

흐름은 대략 이렇다.

1. 일단 돌아가게 만든다.
2. 실제 장애를 몇 번 겪는다.
3. 운영에 필요한 옵션/로그/중지/상태판을 붙인다.
4. 그다음에 README, SOP, CONFIG, BUGFIX LOG를 정리한다.

이건 매우 현실적인 접근이다.  
문서를 먼저 쓰면 실제 운영 흐름과 어긋날 가능성이 높다.  
이 사용자는 먼저 살아있는 프로세스를 만든 뒤, 그 살아있는 흐름을 문서화했다.

## 14. 브랜딩과 이름 변경도 사고과정의 일부였다
마지막 단계에서 이 사용자는 단순 기능을 넘어 프로젝트 정체성도 정리하기 시작했다.

예를 들면:

- 프로젝트명 정리
- README/DEV의 메시지 정리
- run 스크립트 이름을 `run_virtshift.sh`로 변경

이건 단순 꾸미기가 아니다.  
이 시점에는 이미 도구가 "개인용 스크립트 묶음"에서 "다른 사람에게 설명 가능한 실행 엔진"으로 넘어가고 있었다는 뜻이다.

즉, 이름을 바꾸는 행위는 구조가 어느 정도 안정화되었고, 이제 공유 가능한 단위가 되었다는 신호다.

## 15. 사용자의 사고과정을 단계별로 복원하면
이 사용자의 사고 흐름을 더 세밀하게 복원하면 대략 아래와 같다.

### 단계 1. 당장 막히는 것을 해결한다
처음에는 "이걸 만들자"가 아니라 "지금 여기서 막힌다"가 출발점이다.

- git 연결
- 인증
- 현재 폴더 구조
- xml 경로

즉, AI를 개발 착수 도구가 아니라 작업 착수 도구로 사용했다.

### 단계 2. 실행 경로를 운영 표준에 맞춘다
실행이 되기 시작하면 곧바로 운영 표준화 요구가 들어간다.

- `/data` 기반 경로
- conf 분리
- 절대경로
- CSV 매핑

즉, 개인 테스트용이 아니라 실환경에서 팀이 돌릴 수 있는 모양으로 맞추기 시작한다.

### 단계 3. 기능보다 실패 케이스를 먼저 만난다
도구가 동작하면 바로 실패가 드러난다.

- lv down 실패
- import 실패
- ssh/passwd 실패
- precheck 오작동
- lock 충돌
- 맥 중복

이때 중요한 점은, 사용자가 실패를 불평으로 끝내지 않고 구조 개선 요구로 바꿨다는 것이다.

### 단계 4. 운영 도구에 필요한 제어면을 붙인다
이 단계에서 단순 자동화는 운영 도구로 바뀐다.

- nohup
- stop
- stale 정리
- 중복 방지
- status
- result log

즉, "잘 실행되는 것"에서 "문제 생겨도 계속 굴릴 수 있는 것"으로 관심이 이동한다.

### 단계 5. 여러 대를 동시에 본다
여기서 이 프로젝트의 핵심 가치가 드러난다.

- 여러 호스트에서 동시에 돌아가는 작업을 한눈에 보고 싶어 함
- 개별 로그가 아니라 fleet 관점 status를 원함
- VM 개수, 호스트, 타깃, 진행률, 최근 상태 같은 집계 관점을 요구함

즉, 이 사용자의 머릿속 모델은 이미 "VM 1대 처리 스크립트"가 아니라 "멀티 노드 배치 운영판"이었다.

### 단계 6. 이제 다른 사람도 쓰게 만든다
마지막에는 README, USAGE, CONFIG, DEV, SOP, Confluence 정리까지 간다.

즉, 사고의 종착점은 단순 자동화가 아니라 공유 가능한 운영 자산이다.

## 16. 다른 사람이 따라 하려면 무엇을 배워야 하나
이 사용 방식을 다른 사람에게 전파하려면 아래 포인트를 강조하면 좋다.

### 16.1 처음부터 완벽하게 설명하려고 하지 말 것
처음에는 지금 가장 불편한 문제 하나만 정확히 던져도 된다.

좋은 시작 예시는 이런 유형이다.

- "이 경로를 이렇게 바꾸고 싶다"
- "이 단계는 타깃에서 실행되어야 한다"
- "이 로그가 왜 안 남는지 보자"
- "중복 실행되지 않게 해달라"

### 16.2 말로 설명이 길어지면 바로 증거를 붙일 것
스크린샷, 로그, tail, XML 조각, 수동 명령 결과를 바로 붙이면 AI의 추론 범위가 훨씬 좁아진다.

### 16.3 구현법보다 원하는 동작을 먼저 말할 것
예:

- "conf로 빼"
- "로그는 한 번에 봐야 한다"
- "기본은 CSV고 수동도 가능해야 한다"

이런 표현은 구현을 열어두면서 결과는 명확히 만든다.

### 16.4 한 번에 큰 요구보다 작은 요구를 연속으로 줄 것
작은 요청은 검증이 쉽고, 무엇이 좋아졌는지 확인하기 쉽다.

### 16.5 실제로 돌려보고 다시 말할 것
AI 협업의 품질은 첫 답변보다 두 번째, 세 번째 피드백에서 올라간다.  
이 사용자는 거의 매번 실제 실행 결과를 다시 가져왔다.

### 16.6 "더하기"만 하지 말고 "빼기"도 계속 요구할 것
좋은 운영 도구는 기능이 많은 도구가 아니라, 필요한 것만 남은 도구다.

## 17. 이 사용 패턴의 강점
이 방식의 강점은 매우 뚜렷하다.

- 현실 문제에서 출발하므로 바로 쓸 수 있다.
- 증거 기반이라 디버깅 효율이 높다.
- 작은 반복이라 실패 비용이 작다.
- 운영 표준으로 빠르게 수렴한다.
- 기능 추가, 안정화, 문서화, 브랜딩까지 자연스럽게 이어진다.

즉, AI를 "문서 생성기"가 아니라 "실행-검증-정제 루프의 가속기"로 쓴 것이다.

## 18. 이 사용 패턴의 약점 또는 주의점
좋은 점만 있는 것은 아니다. 다른 사람이 따라 할 때는 아래도 같이 알아두면 좋다.

- 요구사항이 매우 빠르게 바뀌므로, 중간중간 기준 문서가 없으면 AI가 이전 방향을 계속 끌고 갈 수 있다.
- 짧은 문장이 많을수록, 맥락을 AI가 잘못 잡는 순간 엇갈릴 수 있다.
- 실환경에서 바로 검증하는 방식이라, stop/log/lock 같은 안전장치가 초기에 없으면 위험할 수 있다.
- 문서화가 너무 늦으면, 이미 만든 구조를 다시 설명하느라 비용이 커진다.

즉, 이 방식은 강력하지만, 중간중간 "지금 기준은 이거다"를 다시 묶어주는 습관이 있으면 더 좋아진다.

## 19. 가장 본받을 만한 점
이 전체 흐름에서 가장 배울 만한 점은 아래 4가지다.

1. **AI를 실행 파트너처럼 썼다**  
상담 상대가 아니라 같이 손을 움직이는 엔지니어처럼 다뤘다.

2. **문제가 생길 때마다 바로 구조 요구로 연결했다**  
실패를 단순 오류가 아니라 설계 개선의 근거로 썼다.

3. **운영 기준을 끝까지 놓치지 않았다**  
경로, 로그, 기본값, 실행 위치, 중복 방지, 상태판 같은 운영 요소를 계속 우선했다.

4. **최종 목표를 "코드 완성"이 아니라 "공유 가능한 운영 자산"으로 봤다**  
그래서 결국 README, SOP, CONFIG, DEV, Confluence, 브랜딩까지 이어졌다.

## 20. 다른 사람에게 공유할 때 이렇게 설명하면 좋다
이 사례를 다른 사람에게 설명할 때는 이렇게 요약하면 된다.

"AI에게 일을 잘 시키려면 처음부터 완성된 설계를 줄 필요는 없다.  
대신 실제 운영에서 막히는 문제를 하나씩 던지고, 실행 결과를 다시 가져오고, 그때그때 기준을 더해가면 된다.  
그러면 AI는 단순 코드 생성기가 아니라, 운영 도구를 함께 다듬는 페어 엔지니어처럼 쓸 수 있다.  
특히 멀티 VM 동시 처리, 병렬 변환, 로그/상태판/중지/락 같은 운영 요소를 붙여가면, 스크립트 묶음이 아니라 배치 운영 엔진으로 성장시킬 수 있다."

## 21. 추가 질문이 필요했는가
이번 분석 문서를 만드는 데 추가 질문은 필수는 아니었다.  
이미 전체 대화 흐름 안에 사용자의 사고과정, 요구사항 진화, 우선순위 변화가 충분히 드러나 있었기 때문이다.

다만 후속 버전을 더 정교하게 만들고 싶다면 아래 질문은 도움이 될 수 있다.

1. 이 도구를 만들 당시 실제로 가장 아까웠던 시간은 무엇이었는가
2. 가장 자주 반복된 수작업은 무엇이었는가
3. AI가 가장 도움 됐다고 느낀 순간은 디버깅, 구조화, 문서화 중 어디였는가
4. 사람에게 시켰으면 오래 걸렸을 일을 AI에게 맡겨서 빨라진 대표 사례는 무엇이었는가

하지만 1차 분석 문서로서는 현재 정보만으로도 충분히 의미 있는 수준이다.
