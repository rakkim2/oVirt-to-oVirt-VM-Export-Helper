# 사용자 프롬프트 패턴 정리

## 1. 문서 목적
이 문서는 실제 대화에서 사용자가 AI에게 어떤 식으로 프롬프트를 던졌는지를 정리한 문서다.

중심은 아래 4가지다.

1. 사용자가 어떤 표현으로 AI에게 일을 시켰는가
2. 어떤 시점에 어떤 종류의 프롬프트가 등장했는가
3. 짧은 프롬프트 안에 어떤 의도가 숨어 있었는가
4. 다른 사람이 비슷하게 쓰려면 어떤 식으로 따라 하면 되는가

즉, 이 문서는 "무슨 기능을 만들었는가"보다 "어떻게 요청했는가"를 정리한 자료다.

## 2. 전체 인상
이 대화에서의 프롬프트는 전반적으로 아래 특징을 가진다.

- 짧다
- 명령형이다
- 실제 운영 문제 중심이다
- 한 번에 하나씩 수정하게 만든다
- AI 답변보다 실행 결과를 다시 던지면서 요구를 진화시킨다
- 기능 추가만큼 삭제, 단순화, 이름 정리 요청이 많다

즉, 설명형 프롬프트보다 "현업 지시형 프롬프트"에 가깝다.

## 3. 프롬프트의 큰 유형
전체 대화에서 사용된 프롬프트는 대략 아래 9개 유형으로 나눌 수 있다.

1. 연결/환경 착수형
2. 구조/경로 변경형
3. 옵션/설정 정리형
4. 실행 제어형
5. 장애 제보형
6. 관측/로그 강화형
7. 단순화/리팩토링형
8. 문서화/공유형
9. 메타 분석형

아래부터는 이 9개를 더 자세히 정리한다.

## 4. 연결/환경 착수형 프롬프트
초반 프롬프트는 거창한 설계보다 "일단 붙여라" 성격이 강했다.

대표 예시:

- `ovirt to ovirt exporter 폴더에 있는거 깃 연결해봐.`
- `내가 커밋 요청 할수도 있나`
- `음 pull request 안왔는데 해당 git으로`
- `인증 어케함`
- `ㅇㅇ 했음. 너랑 근데 어케 연결되는거야. gh 모듈이. 너가 내 터미널을 써서 그런가?`

이 유형의 특징:

- AI를 코드 생성기보다 작업 착수 파트너로 사용한다.
- "설명"보다 "바로 연결/실행"을 먼저 요구한다.
- 사용자는 도구 연결 구조를 이해하려고 하지만, 기술적 세부 설명보다 실사용 가능 여부를 더 중시한다.

숨은 의도:

- 이 AI가 실제로 git/github/terminal 맥락에서 얼마나 함께 움직일 수 있는지 확인
- 이후 커밋, PR, 브랜치 작업까지 맡길 수 있는지 탐색

다른 사람이 따라 쓸 수 있는 템플릿:

```text
이 폴더 기준으로 git 연결부터 해봐.
이후 커밋/브랜치/PR도 내가 요청할 수 있는지 같이 보자.
```

## 5. 구조/경로 변경형 프롬프트
가장 많이 반복된 유형 중 하나다.  
사용자는 경로, 디렉터리 배치, 산출물 위치에 매우 민감했다.

대표 예시:

- `xml만들때 폴더가 지금 v2v 현재 폴더 밑에다가 xml 폴더 내에 넣거든. 그러지 말고 /data 폴더 밑에 넣어보자.`
- `conf는 그냥 지금 폴더에 다시 빼. 그리고 import.conf 파일로 이름 변경하고`
- `그래. 근데 지금 new project 폴더 위에 있는데, codex project로 모든걸 이동할 수 잇니`
- `그리고 굳이 ova, v2v폴더를 나눌 필요가 있나? 그럼 복잡한데.. 그냥 하나로 하자.`
- `기본 경로는 /data/xml로 해라.`
- `그리고 config 수정을 추가적으로 하자. 여기서 아예 BASE_DIR을 /data와 같이 설정해서.`

이 유형의 특징:

- 구현 내부보다 결과 산출 경로를 먼저 통제한다.
- 운영 표준 경로(`/data`, `/data/script`, `/data/xml`, `/data/v2v_log`)를 중심축으로 세운다.
- "이 구조가 복잡하다"는 판단이 나오면 다시 합치거나 줄인다.

숨은 의도:

- 개인 테스트 스크립트가 아니라 운영 표준 경로에 맞는 도구로 만들기
- 나중에 다른 사람도 경로를 쉽게 예측할 수 있게 하기

다른 사람이 따라 쓸 수 있는 템플릿:

```text
산출물 경로를 운영 표준으로 맞춰줘.
지금 구조는 복잡하니 /data 기준으로 다시 단순화해줘.
```

## 6. 옵션/설정 정리형 프롬프트
기능 구현과 거의 동시에 계속 등장한 유형이다.  
사용자는 "하드코딩을 줄이고 conf로 빼는 것"을 매우 중요하게 봤다.

대표 예시:

- `링크나 디렉토리 같은거나 뭐 옵션같은거는 conf 로 만들어`
- `conf는 그냥 지금 폴더에 다시 빼.`
- `global 옵션은 젤 위에 놓고.`
- `쉘스크립트 별로 따로 넣을 옵션은 따로 밑에 주석으로 스크립트명 쓴다음에 넣어.`
- `config 이름도 수정하자. 일반적이게 ... 뭐라해야하나 ..?`
- `config 내의 중복 요소는 없나? 정리하고 싶은데`
- `기본 값인 것들은 다 주석처리해놓아야지.`
- `꼭 적어야하는것만 주석 풀어놔.`
- `다른 스크립트에서도 저거를 가지고 기본값을 보게 하고있어?`

이 유형의 특징:

- 설정 파일을 단순 변수 덤프가 아니라, 사람이 읽고 바로 이해하는 운영 가이드로 다룬다.
- "필수/선택", "전역/스크립트별", "주석 처리 여부"까지 요구한다.
- 단순 기능보다 설정 UX를 신경쓴다.

숨은 의도:

- 나 말고 다른 운영자도 conf만 보면 바로 쓸 수 있게 만들기
- 하드코딩을 제거해 환경 이동성을 확보하기

다른 사람이 따라 쓸 수 있는 템플릿:

```text
하드코딩은 전부 conf로 빼고,
필수값은 위에, 선택값은 주석으로 아래에 정리해줘.
사람이 바로 읽을 수 있게 설명도 붙여줘.
```

## 7. 실행 제어형 프롬프트
이 유형은 도구를 "실행 가능한 운영 엔진"으로 바꾸는 데 핵심 역할을 했다.

대표 예시:

- `run qemu 할때 virsh -r list로 해당 vm이 있으면 실행이 안되는거로 하자.`
- `nohup으로 돌게해. 그리고. 같은 vm은 중복 실행되지 않게 해볼래`
- `락은 어떻게 걸었어?`
- `중간에 실행이 갑자기 꺼지면, 락이 있는 상태인데, 그럴때는 어떻게 하나/`
- `run qemu의 설정에는 ova를 실행할지 말지를 옵션을 더줘.`
- `run import는 import_v2v_remote로 바꿔.`
- `import v2v는 옵션을 다시 보자.`
- `import v2v는 옵션을 --실행위치=타겟SPM or remote를 강제해야지 돌아가게 해.`
- `import 를 stop하는 방법 ?`

이 유형의 특징:

- 실행 시작 조건, 중복 실행, 실행 위치, 중지 방법을 계속 다듬는다.
- "잘 돌아가는 기능"보다 "운영자가 제어 가능한 도구"가 되게 만든다.

숨은 의도:

- 여러 VM, 여러 호스트를 동시에 돌릴 때 사고 나지 않게 만들기
- 실행/중단/재시작이 운영자 입장에서 명확해야 함

다른 사람이 따라 쓸 수 있는 템플릿:

```text
이 스크립트를 운영용으로 쓸 거라서,
중복 실행 방지, stop, nohup, 실행 위치 강제까지 같이 넣어줘.
```

## 8. 장애 제보형 프롬프트
이 대화에서 가장 강력했던 유형이다.  
사용자는 이론보다 실제 증상을 그대로 AI에 던졌다.

대표 예시:

- `그래. 지금 근데 lv down 되는 단계거든. 근데 이상태 fail 실패하네`
- `수동 명령어`
- `지금 다 잘되다가 갑자기 1개 vm이 import해도 부트디바이스 못찾네.`
- `remote 하면 왜 로그가 안남지 흠. remote 제대로 되는거 맞아? 수동으로 실행하면?`
- `뭐야 이거 수동으로 실행해도 ssh 입력하라 나오네`
- `precheck 하는게 지금 현재 로컬이 아니라 타겟의 로컬에서 실행하는거지?`
- `지원 하는데. precheck 로직이 잘못된거 아냐?`
- `이런식으로 나오는데 이건 뭐지`
- `import 로그 없음`
- `부모프로세스 하위 다 죽이기 명령어 ?`

이 유형의 특징:

- 현상 그대로 던진다.
- "왜 그런가?"와 동시에 "지금 뭘 해야 하나?"를 묻는다.
- 추상적인 설명보다, 실제 로그와 증거가 같이 온다.

숨은 의도:

- AI에게 이론 설명이 아니라 현장 디버깅을 맡기기
- 재현 가능한 단서만 던져서 바로 원인 좁히기

다른 사람이 따라 쓸 수 있는 템플릿:

```text
지금 이 단계에서 이렇게 실패한다.
아래는 실제 로그/명령 결과다.
원인 추정 말고, 바로 확인해야 할 지점부터 봐줘.
```

## 9. 관측/로그 강화형 프롬프트
사용자는 로그를 단순 기록이 아니라 "판단 도구"로 봤다.  
그래서 로그/상태판 관련 프롬프트가 매우 많이 나왔다.

대표 예시:

- `각 쉘 스크립트의 로그도 남겼으면 좋겠는데`
- `로그는 명령어 쓴 거에 굉장히 자세히 나오는 출력이여야하는거알지?`
- `status board log 가 기록 안되는데?`
- `show할때 ova로 마는 것 같은 다른 상태는 안보이잖아.`
- `퍼센트 현황도 나오게.`
- `log 중에 현재 상태만 보는 걸로 만들어줘.`
- `한번에 보는 현황판 로그는 하나로 보일 수 있도록`
- `결과 로그도 전체집계로 더 만들어.`
- `all show status`
- `각각 볼때 앞 줄에 호스트명이 보이게 해줘.`
- `vm 개수 번호좀 붙이자. 그리고 타겟 호스트도 적어줄래`

이 유형의 특징:

- 로그를 남기기만 하면 안 되고, 읽히고 집계되고 판단 가능해야 한다.
- "개별 로그", "결과 로그", "상태판", "fleet 단위 집계"를 분리해서 생각한다.
- 운영자가 여러 대를 동시에 볼 상황을 계속 전제로 둔다.

숨은 의도:

- 멀티 실행 환경에서 어느 VM이 어디까지 갔는지 한 화면에서 알고 싶음
- 실패 원인을 빠르게 찾아 MTTR을 줄이고 싶음

다른 사람이 따라 쓸 수 있는 템플릿:

```text
로그를 남기는 것만으로는 부족하고,
운영자가 한 화면에서 현재 상태와 실패 원인을 바로 볼 수 있게 만들어줘.
```

## 10. 단순화/리팩토링형 프롬프트
이 사용자는 AI가 구조를 늘리면 다시 줄이는 요청을 자주 했다.

대표 예시:

- `굳이 ova, v2v폴더를 나눌 필요가 있나?`
- `import conf는 딱히 필요 없다. 지우고`
- `run_import 나 import sh 둘중 하나만 냅둬.`
- `모든 스크립트에 대해 리팩토링을 진행해.`
- `내가 아까 너한테 보여준 import 초기버전처럼 짧게는 못해?`
- `아니야 파일 더 만들지마 그냥 하나에서 해...`
- `폴더 정리해 불필요한 파일`
- `이름을 그냥 env 로 바꿀까 ?`

이 유형의 특징:

- 기능 추가 뒤에 곧바로 "근데 이건 너무 길다/많다/복잡하다"가 따라온다.
- 구조적 군더더기를 제거하면서 운영자 시점의 단순성을 되찾으려 한다.

숨은 의도:

- 자동화 도구가 커져도 실제 쓰는 사람 입장에서 어려워지면 안 됨
- 스크립트 수, 설정 수, 로그 수를 최소화하려는 성향

다른 사람이 따라 쓸 수 있는 템플릿:

```text
기능은 유지하되 구조가 너무 복잡해졌다.
파일 수, 설정 수, 로그 수를 줄이는 방향으로 리팩토링해줘.
```

## 11. 문서화/공유형 프롬프트
기능이 어느 정도 안정화된 뒤에는 문서 요청이 강해졌다.

대표 예시:

- `사용법 쉽게 업데이트 해.`
- `중요한 것만 업뎃.`
- `초보자가 따라할 수잇게 sop 만든다 생각해`
- `read.me 를 sop로 거의 치환하고`
- `그외 상세 내역들은 conf.md로 따로 빼`
- `컨플에 올려줘 필요한 것만. 사람들에게 공유할것만. 중요한것만`
- `이 페이지에서의 대화들은 내가 ai코딩을 처음해보게 된 건데, 어떤 식으로 내가 요청했었는지...`

이 유형의 특징:

- 문서의 독자까지 명확하다.  
  예: 초보자, 운영자, 팀원, 인수인계 대상, 공유 대상
- README, SOP, CONFIG, DEV, CONFLUENCE를 역할별로 나눈다.

숨은 의도:

- 이 도구가 개인 메모가 아니라 팀 자산이 되길 원함
- "돌아가는 코드"를 "설명 가능한 체계"로 승격시키려 함

다른 사람이 따라 쓸 수 있는 템플릿:

```text
이제 다른 사람이 그대로 따라 쓸 수 있게 문서로 정리해줘.
SOP, 설정 가이드, 개발 이력을 역할별로 나눠줘.
```

## 12. 이름/브랜딩/정체성 정리형 프롬프트
마지막 단계에서는 프로젝트 이름과 메시지까지 정리하기 시작했다.

대표 예시:

- `run_qemu_pipeline.sh 을 모두 run_virtshift.sh 로 바꾸자.`
- `readme 프로젝트 설명과 DEV의 설명을 잘 써줬는데, 이게 뭘 의미하는지에 대해 중점적으로 쓰지 않았어.`
- `제일 중요한건 멀티가 가능하고, 이렇게 써지는게 중요한거야. 다시 정리해봐`

이 유형의 특징:

- 단순 코드 정리가 아니라, 프로젝트가 무엇을 의미하는지까지 통제한다.
- 이름, 설명, 핵심 메시지를 맞춰서 외부 공유 가능 상태로 만든다.

숨은 의도:

- 이 프로젝트를 "스크립트 모음"이 아니라 "실행 엔진"으로 보이게 하기
- 핵심 가치인 멀티 실행/병렬 처리 관점을 전면에 올리기

다른 사람이 따라 쓸 수 있는 템플릿:

```text
이제 기능 설명보다 프로젝트 의미가 먼저 보이게 정리해줘.
이 도구의 핵심 가치를 이름과 README에 반영해줘.
```

## 13. 메타 분석형 프롬프트
대화 후반부에는 아예 "AI와 협업한 방식 자체"를 분석 대상으로 삼았다.

대표 예시:

- `내가 ai코딩을 처음해보게 된 건데, 어떤 식으로 내가 요청했었는지. 처음부터 분석해봐.`
- `이것 말고,. 내가 너한테 프롬프트 했던 것들을 정리해봐.`

이 유형의 특징:

- 산출물만이 아니라 사용 방식 자체를 자산화하려 한다.
- 다른 사람이 따라 할 수 있도록 협업 패턴을 복기하려 한다.

숨은 의도:

- 나의 요청 방식이 어떤 구조를 가졌는지 스스로 파악
- 이를 교육 자료/가이드로 재사용

## 14. 프롬프트 문장 스타일 자체의 특징
문체만 따로 보면 아래 특징이 반복된다.

### 14.1 짧은 명령형
예:

- `바꿔`
- `지워`
- `올려`
- `커밋`
- `메인에 pr`

의미:
- 장황한 설명 없이 바로 작업 모드로 전환시키는 신호

### 14.2 방향 전환형
예:

- `다시.`
- `아니야.`
- `그럼 어쩌라고`
- `이런식으로 하는게 맞아?`

의미:
- 이전 답변이나 구현 방향이 마음에 들지 않을 때 즉시 리셋
- AI가 이전 가정에 묶이지 않게 끊어주는 역할

### 14.3 운영 기준 삽입형
예:

- `실행중인 호스트에서 실행하면 안돼. 타겟에서 실행해야해.`
- `로그는 명령어 쓴 거에 굉장히 자세히 나오는 출력이여야하는거알지?`
- `기본은 import가 아니고 ova 만들어지는건가?`

의미:
- 기술 제안보다 운영 원칙을 상위 제약으로 두는 방식

### 14.4 비교/의심 제기형
예:

- `이건가?`
- `이거는 맞나?`
- `왜 안되지 ...`
- `precheck 로직이 잘못된거 아냐?`

의미:
- AI의 설명을 수동 검증하며 계속 압박 테스트

## 15. 요구가 추가된 방식
이 대화에서 요구사항은 한 번에 정리되지 않았고, 대체로 아래 패턴으로 추가되었다.

1. 기능 요청
2. 실행 결과 확인
3. 실제 실패 발견
4. 운영 제약 추가
5. 구조 단순화
6. 문서화

예를 들면:

- 처음에는 `xml 저장 위치 변경`
- 다음에는 `conf로 분리`
- 다음에는 `옵션 이름 정리`
- 다음에는 `nohup/lock/stop`
- 다음에는 `상태판/로그`
- 다음에는 `fleet 수준 관측`
- 마지막에는 `README/DEV/브랜딩`

즉, 프롬프트는 단절된 요청들의 모음이 아니라, 계속 진화하는 운영 요구의 흔적이었다.

## 16. 이 프롬프트 방식의 장점
이 방식이 강한 이유는 다음과 같다.

- 실제 문제에서 출발하므로 결과물이 바로 쓸 만하다.
- 요구가 짧아도 기준이 선명해서 AI가 빠르게 작업할 수 있다.
- 증거를 자주 붙여서 디버깅 정확도가 높다.
- 작은 단위로 계속 수정하므로 실패 비용이 낮다.
- 기능 추가, 안정화, 문서화, 브랜딩까지 자연스럽게 연결된다.

## 17. 다른 사람이 그대로 따라 하려면
이 사례를 다른 사람이 따라 하려면 아래 순서가 좋다.

1. 처음에는 아주 구체적인 문제 하나만 던진다.
2. 기능이 생기면 바로 실환경에서 돌려본다.
3. 실패하면 로그/증거를 붙여 다시 던진다.
4. 불편하면 구조를 단순화하라고 다시 말한다.
5. 어느 정도 안정화되면 README/SOP/CONFIG로 정리시킨다.
6. 마지막에는 이름과 프로젝트 메시지까지 정리한다.

## 18. 다른 사람이 바로 따라 쓸 수 있는 프롬프트 골격

### 18.1 기능 추가 요청
```text
지금 이 단계에서 필요한 건 이것 하나다.
현재 동작은 A인데, B처럼 바꿔줘.
```

### 18.2 장애 분석 요청
```text
지금 실제로 이렇게 실패한다.
아래 로그/스크린샷 기준으로 원인부터 좁혀줘.
```

### 18.3 운영화 요청
```text
기능은 됐고, 이제 운영용으로 써야 한다.
nohup, stop, lock, 로그, 상태판까지 붙여줘.
```

### 18.4 단순화 요청
```text
지금 구조가 복잡하다.
기능은 유지하고 파일/설정/로그 구조는 더 단순하게 줄여줘.
```

### 18.5 문서화 요청
```text
이제 다른 사람이 바로 따라할 수 있게 SOP/설정가이드/개발이력으로 나눠서 정리해줘.
```

## 19. 결론
이 대화에서의 프롬프트는 "긴 설명형 프롬프트"가 아니었다.  
오히려 짧고 단호한 운영 지시형 프롬프트, 실증 기반 장애 제보형 프롬프트, 구조 단순화 프롬프트가 핵심이었다.

그리고 가장 중요한 점은, 사용자가 AI를 한 번 요청하고 끝내지 않았다는 것이다.  
실행 결과를 다시 넣고, 제약조건을 추가하고, 이름을 바꾸고, 문서를 만들면서 AI를 점점 더 깊게 작업 파트너처럼 사용했다.

즉, 이 문서에서 정리한 것은 단순한 "질문 모음"이 아니라, 실제 운영 도구를 AI와 같이 만들어가는 프롬프트 습관의 패턴이다.
